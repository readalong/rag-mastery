01 — RAG ARCHITECTURESThe Nine Types You Must Know

Each RAG variant solves a specific weakness. The trick isn't choosing the most advanced one — it's choosing the right one for your use case.

📄
Naive RAG
Baseline

The simplest form: chunk → embed → store → retrieve → generate. No reranking, no query rewriting, no validation. Your documents go into a vector DB, queries get embedded, top-k similar chunks come back, and the LLM generates from them.

Where it works: Small, clean datasets. Internal FAQs. Simple document Q&A where the corpus is tightly scoped and high quality.

⚠ Known Failures

High noise risk — irrelevant chunks confuse the LLM. No self-correction; if retrieval fails, you get confident hallucination. Struggles with multi-part questions or ambiguous queries.

💡 Practitioner Tip

Naive RAG is your starting baseline, never your destination. Build it first, measure it, then layer improvements. You can't optimize what you haven't benchmarked.

🔬
Advanced RAG
Production-Ready

Adds pre-retrieval and post-retrieval intelligence to the naive pipeline. Think query rewriting, hybrid search, reranking, and context compression layered before and after the core retrieval step.

Key additions: Query routing or transformation happens before retrieval. Reranking with cross-encoders happens after retrieval. Context is compressed to remove noise before generation.

🧩
Modular RAG
Composable

Breaks the RAG pipeline into swappable, composable modules. Each component — chunker, retriever, reranker, generator — can be independently upgraded, tested, and configured per use case.

Why it matters: Real-world systems need different strategies for different document types. Legal docs need different chunking than code. Product manuals need different retrieval than research papers. Modular RAG lets you mix and match.

💡 Practitioner Tip

Design your RAG system with clean interfaces between modules from day one. Even if you start naive, having pluggable retriever/reranker/generator interfaces means you can upgrade incrementally without rewriting the whole pipeline.

🪞
Self-RAG
Self-Reflective

The model self-critiques its own generation. After producing an answer from retrieved documents, it evaluates whether the answer is actually supported by the source material and adjusts if discrepancies are found.

How it works: The LLM generates special "reflection tokens" at each step. It decides: (1) Do I even need retrieval for this? (2) Is the retrieved content relevant? (3) Does my answer actually follow from the evidence? (4) Is the answer useful?

// Self-RAG pseudocode flow
function selfRAG(query) {
  if (needsRetrieval(query)) {
    docs = retrieve(query)
    answer = generate(query, docs)
    if (!isSupported(answer, docs)) {
      return regenerate(query, docs) // retry with critique
    }
  } else {
    answer = generate(query) // answer from parametric knowledge
  }
  return answer
}
⚠ Trade-off

Self-RAG adds latency — every answer gets a self-critique pass. It's best when accuracy is more important than speed, such as in medical, legal, or financial applications.

🔧
Corrective RAG (CRAG)
Validation Layer

Adds a critical validation layer after retrieval. A lightweight evaluator scores documents as "Correct," "Incorrect," or "Ambiguous." If retrieval quality is poor, CRAG triggers corrective actions — query transformation, web search fallback, or knowledge strip decomposition.

Key differentiator from Self-RAG: CRAG uses a separate, often smaller model as the evaluator, making it more modular. You can evolve the checker independently from the generator.

# CRAG flow
Query → Retrieve → Grade Documents
  ├─ Relevant? → Yes → Generate Answer
  └─ Not sure / No?
       ├─ Transform Query → Re-retrieve
       └─ Web Search Fallback → Generate
💡 Practitioner Tip

CRAG is ideal for compliance-critical systems. Use a fine-tuned T5 or similar small model as the grader — it's fast and cheap. The key insight: decomposing documents into "knowledge strips" and grading each strip individually catches partial relevance that chunk-level grading misses.

🕸️
Graph RAG
Relationship-Aware

Uses a knowledge graph to understand how entities and concepts are connected. Instead of just matching text similarity, Graph RAG traverses relationships — finding relevant information even when there's no lexical or semantic overlap with the query.

Power domains: Pharma (drug interactions), finance (regulatory dependencies), investigative journalism (hidden connections), HR (org structures), cybersecurity (attack chains).

💡 Practitioner Tip

Graph RAG shines when your queries are about relationships, not just facts. "Which employees report to managers who approved the Q3 budget?" requires understanding org structure — vectors alone won't solve this. Pair it with Neo4j or similar for the graph layer.

🤖
Agentic RAG
Autonomous

Instead of a fixed pipeline, an LLM-powered agent decides what to retrieve, when to retrieve, and whether the retrieval was sufficient. It can plan multi-step research, use tools, backtrack, and refine its approach.

Sub-patterns:

Single-Agent: One agent manages retrieval, routing, and generation. Good for systems with limited tools or data sources.

Multi-Agent: Specialized agents for different roles — one retrieves, one validates, one synthesizes. Enables parallel processing and handles diverse query types.

Adaptive: Dynamically adjusts strategy based on query complexity. Simple questions get fast retrieval; complex ones trigger multi-step reasoning.

⚠ Complexity Warning

Agentic RAG is powerful but hard to debug. Each agent call adds latency and cost. Start with single-agent, prove value, then decompose into multi-agent only if you have genuinely distinct retrieval domains.

🔀
Hybrid RAG
Multi-Strategy

Combines multiple retrieval strategies in one system — typically dense vector search + sparse keyword search (BM25), merged via Reciprocal Rank Fusion (RRF). The idea: vectors catch semantic meaning, keywords catch exact terms.

Why both: Pure vector search misses product IDs, ticket numbers, error codes, version tags. Pure keyword search misses paraphrased intent. Hybrid catches both.

💡 Practitioner Tip

Teams mixing keyword and vector search see double-digit gains in relevance without sacrificing latency — this is low-hanging fruit most teams still ignore. Start with a 0.5 alpha (equal weighting) and tune from there. For enterprise data with lots of identifiers, bias toward BM25 (alpha 0.6–0.7).

🖼️
Multimodal RAG
Beyond Text

Integrates image, audio, tabular, and video embeddings into retrieval. A maintenance engineer can ask "Show me the failure pattern for turbine blade anomalies" and the system retrieves technical diagrams, sensor logs, and text reports together.

Architecture: Query (text/image/audio) → Retriever (multi-modal embeddings) → Top hits across modalities → Generator (multi-modal model)

💡 Practitioner Tip

Don't try to embed everything into one vector space. Use modality-specific embedders (CLIP for images, Whisper for audio) and merge results at the ranking stage. This avoids the "jack of all trades, master of none" problem.

💡 The Golden Rule

You can't fix a mis-framed question later in the pipeline. Treat every user query as raw material to be refined — query processing is the "front door" of your architecture.

The Two Phases

Indexing (Offline)

Happens once (or on a schedule). Load documents → clean & parse → chunk → enrich with metadata → embed → store in vector DB + keyword index. This is where you invest in data quality.

💡 Tip

Index-time is cheap, query-time is expensive. Front-load as much work as possible into indexing — contextual headers, metadata extraction, pre-computed summaries. Every dollar spent here saves ten at query time.

Querying (Online)

Happens per request. Process user query → transform/expand → retrieve from index → rerank results → compress context → generate response → (optional) validate output.

💡 Tip

Set a latency budget. Decide upfront: are you optimizing for sub-second responses (chatbot) or can you tolerate 5-10s (research assistant)? This shapes every downstream choice.

03 — CHUNKING STRATEGIESHow You Split Changes Everything

Chunking is the most underrated part of RAG. Bad chunking poisons everything downstream — no amount of reranking fixes context ripped from its meaning.

Split text into fixed token windows (e.g., 512 tokens) with overlap (e.g., 50-100 tokens). The simplest approach.

When to use: Homogeneous content, initial prototyping, when you need predictable chunk sizes for cost estimation.

# LangChain example
from langchain.text_splitter import TokenTextSplitter
splitter = TokenTextSplitter(
    chunk_size=512,
    chunk_overlap=50
)
⚠ Pitfall

Fixed-size splits mid-sentence and mid-paragraph, destroying context boundaries. A paragraph about "Q2 revenue" may get split so the number is in one chunk and the context is in another.

Tries to split on natural boundaries in order: paragraph breaks → sentence breaks → word breaks → character breaks. Only falls to the next level if chunks are still too large.

When to use: General-purpose. Good default for most text documents. Respects natural boundaries while staying within size limits.

# Recommended default separators
separators = ["\n\n", "\n", ". ", " ", ""]
💡 Tip

This is the sweet spot for most teams. Paragraph-level splitting balances context and precision, improving recall by 15-25% per LangChain evaluations.

Uses embeddings to detect topic shifts. Sentences with high semantic similarity stay grouped; a significant drop in similarity triggers a new chunk boundary.

When to use: Technical documentation, product manuals, long-form research where topics weave in and out.

💡 Tip

Semantic chunking costs more (requires embedding every sentence), but pays off for complex content. For simpler content like FAQs or short articles, fixed-size chunks work fine — don't over-engineer.

Respects the document's own structure — headings, sections, tables, code blocks. Splits at structural boundaries rather than arbitrary positions.

When to use: Markdown docs, HTML pages, technical specifications, legal contracts with numbered sections.

💡 Tip

Parse the document format first (Markdown headers, HTML tags, PDF structure) and use those as primary split points. This preserves the author's intended grouping of information.

Store documents at multiple granularities — small chunks for precise retrieval, large parent chunks for full context. Retrieve on the small chunk, but pass the parent chunk to the LLM.

The insight: Small chunks retrieve with higher precision. But the LLM needs surrounding context to generate good answers. Parent-doc gives you both.

# Hierarchy: Section → Paragraph → Sentence
# Retrieve: match at sentence level
# Return: the full paragraph (or section) containing the match
matched_sentence = retrieve(query, level="sentence")
parent_paragraph = get_parent(matched_sentence)
answer = generate(query, context=parent_paragraph)
💡 Tip

This is one of the highest-impact techniques for reducing hallucination. The small chunk ensures you find the right spot; the parent chunk ensures the LLM has enough context to answer correctly.

Anthropic's Contextual Retrieval: The Chunk Context Problem

Traditional chunking loses context. A chunk might say "The company's revenue grew by 3% over the previous quarter" — but which company? Which quarter? The chunk was ripped from its document context.

Anthropic's solution: Before embedding, use an LLM to prepend a short contextual explanation to each chunk that situates it within the full document. This "contextual header" dramatically reduces retrieval failure.

# Contextual Retrieval: prepend context to each chunk
for chunk in document.chunks:
    context = llm.generate(
        prompt=f"""Given the full document, provide a short context
        for this chunk. Document: {document.full_text}
        Chunk: {chunk.text}"""
    )
    chunk.text = f"{context}\n{chunk.text}"
    chunk.embedding = embed(chunk.text)
💡 Key Numbers

Contextual embeddings alone reduce retrieval failure by 35%. Combined with contextual BM25 and reranking, failures drop by 67%. Use prompt caching to keep costs manageable — the full document is the same across all chunks, so cache it.

⚠ Cost Consideration

You make one LLM call per chunk during indexing (passing the entire document each time). Use a small, cheap model (e.g., Haiku-class) for context generation. With prompt caching, the document tokens are only billed once.

Chunk Size: The Most Debated Hyperparameter

Small chunks (200-400 tokens)

Pros: Higher retrieval precision, less noise, lower token cost per retrieval.

Cons: May lack sufficient context for generation, need more chunks to cover a topic.

Best for: Q&A, factoid retrieval, structured docs.

Large chunks (800-1500 tokens)

Pros: More context per chunk, better for complex reasoning, fewer retrieval calls needed.

Cons: More noise per chunk, higher token cost, lower retrieval precision.

Best for: Summarization, analysis, long-form answers.

💡 The Right Answer

There is no universal optimal chunk size. Test 400, 600, 800, and 1000 on your actual data with your actual queries. Measure precision-recall on a held-out eval set. The "best" size depends on your domain, doc types, and query patterns. GCP's RAG Engine team recommends starting with 512 tokens and 100 token overlap for production, then tuning.

04 — RETRIEVAL TECHNIQUESFinding the Right Needle

Retrieval is the make-or-break stage. If you retrieve the wrong documents, no amount of prompt engineering will save you.

Search Strategies

Embeds queries and documents into the same vector space, then finds nearest neighbors by cosine similarity. Catches semantic meaning and paraphrases well.

Weakness: Fails on exact identifiers — product IDs, error codes, ticket numbers, version tags lose priority in embedding space. Also struggles with negation and very short queries.

💡 Tip

Never use dense search alone in production. It's necessary but not sufficient. Always pair with keyword search for hybrid retrieval.

Classic keyword-based search using term frequency and inverse document frequency. Excels at exact matches, specific terms, identifiers.

Weakness: No semantic understanding. "Car" and "automobile" are completely different terms. Can't handle paraphrasing or intent.

💡 Tip

BM25 is free to run (local computation), requires no GPU, and is surprisingly competitive for queries containing unique identifiers or technical terms. Don't dismiss it as "old school."

Run both dense and sparse search in parallel, then merge results using Reciprocal Rank Fusion (RRF). Each result gets a score based on its rank in each list, and the combined scores determine the final ordering.

# Reciprocal Rank Fusion
def rrf_score(rank, k=60):
    return 1.0 / (k + rank)

# Merge results from vector and keyword search
combined = {}
for rank, doc in enumerate(vector_results):
    combined[doc.id] = combined.get(doc.id, 0) + rrf_score(rank)
for rank, doc in enumerate(keyword_results):
    combined[doc.id] = combined.get(doc.id, 0) + rrf_score(rank)

# Sort by combined score
final_results = sorted(combined.items(), key=lambda x: x[1], reverse=True)
💡 Alpha Tuning

Many vector DBs let you set an alpha weight between dense and sparse. Start at 0.5 (equal weight). For enterprise data with lots of IDs/codes, bias toward BM25 (alpha 0.6-0.7). For conversational queries, bias toward vectors (alpha 0.3-0.4).

Pre-filter your search by structured metadata (date, department, document type, version, language, tenant) before running similarity search. Reduces the search space and ensures relevance.

💡 Tip

Metadata filtering is non-negotiable for multi-tenant systems. Always attach: source document name, page number, section heading, date, version, and any domain-specific tags. Rich metadata is the difference between "I found something related" and "I found the exact section."

Reranking: The Precision Multiplier

After initial retrieval returns a broad set of candidates (top-50 to top-150), a reranker rescores each result against the query with a more powerful model.

Cross-Encoder Reranking

Feeds the query + each candidate chunk through a transformer model (BERT-based) that outputs a relevance score. Very accurate but slower — processes each pair individually.

Best when: Quality matters more than latency.

LLM-as-Reranker

Use the LLM itself to score relevance (e.g., "Rate 1-10 how relevant this passage is to the query"). More flexible but more expensive.

Best when: Domain-specific relevance that off-the-shelf models don't capture.

💡 The "Lost in the Middle" Problem

LLMs pay most attention to the beginning and end of their context window, and least attention to the middle. Reranking solves this by putting the most relevant content first. This single change can dramatically improve answer quality — it's why Google's Vertex AI team considers reranking essential for production RAG.

05 — QUERY PROCESSING & GENERATIONThe Front Door and the Back Door

Smart query processing can outperform larger models on the same data. And generation is where all your retrieval work pays off — or doesn't.

Query Transformation Techniques

HyDE — Hypothetical Document Embeddings

Instead of embedding the short user query directly, ask the LLM to generate a hypothetical answer first, then embed that for retrieval. Since documents in your DB contain answers (not questions), a hypothetical answer is more semantically similar to the actual documents.

# HyDE flow
user_query = "What is the refund policy?"

# Step 1: Generate hypothetical answer
hypothetical = llm.generate(f"Write a detailed answer to: {user_query}")
# → "Our refund policy allows returns within 30 days of purchase..."

# Step 2: Embed the hypothetical answer (not the question)
query_embedding = embed(hypothetical)

# Step 3: Retrieve documents similar to the hypothetical answer
results = vector_search(query_embedding, top_k=10)
⚠ HyDE Caveat

HyDE only works well when the LLM has some knowledge about the topic. If it generates a completely wrong hypothetical, retrieval will go off the rails. Always test with queries where the LLM has zero domain knowledge — if results degrade, add a fallback to standard embedding.

Multi-Query Expansion

Generate multiple reformulations of the user's query, run retrieval for each, and merge the results. Catches documents that match different phrasings of the same intent.

# User asks: "How do I handle errors in async Python?"
# Generated sub-queries:
queries = [
    "Python asyncio exception handling",
    "try except in async await Python",
    "error handling patterns for asynchronous Python code"
]
# Retrieve for each, merge with RRF, deduplicate

Step-Back Prompting

Before retrieving, ask the LLM to rephrase the query at a higher level of abstraction. "What's the GDP of France in 2024?" becomes "What are France's recent economic indicators?" — casting a wider net that catches relevant context.

Query Decomposition

Break complex multi-part questions into simpler sub-questions, retrieve for each independently, then synthesize.

# Complex: "Compare the revenue growth and employee count
#          of Apple and Google over the last 3 years"
# Decomposed:
sub_queries = [
    "Apple revenue growth 2023-2026",
    "Google revenue growth 2023-2026",
    "Apple employee count 2023-2026",
    "Google employee count 2023-2026"
]

Context Compression

After retrieval and reranking, pass the retrieved documents through the LLM to extract only the relevant parts before generating the final answer. This reduces noise and saves tokens.

💡 Tip

Context compression is especially valuable when your chunks are large. Instead of passing 5 full paragraphs, compress each to just the sentences relevant to the query. Less noise = sharper answers + lower token costs.

Generation Best Practices

Always include source attribution. Instruct the LLM to cite which chunks it used. This enables fact-checking and builds user trust.
Set the system prompt to ground answers in retrieved context. Explicitly say: "Answer based ONLY on the provided context. If the context doesn't contain the answer, say so."
Order context by relevance. Place the most relevant chunks first in the prompt, due to the "lost in the middle" effect.
Use structured context delimiters. Wrap each chunk in clear XML-like tags with metadata: <source doc="..." page="...">.
Include a "no answer" escape hatch. The LLM should be able to say "I couldn't find enough information" rather than hallucinating.
Control temperature. Use low temperature (0.0-0.2) for factual Q&A, moderate (0.3-0.5) for synthesis and summarization tasks.

06 — ADVANCED TECHNIQUESThe Techniques That Separate Good From Great

These are the patterns that emerge when teams have graduated past the basics and are optimizing for real-world edge cases.

🏷️
Contextual Headers at Index Time

Prepend document-level metadata to every chunk before embedding: document title, section heading, date, author, summary. The chunk carries its own context wherever it goes.

# Before: chunk is orphaned
chunk = "Revenue grew 3% over the previous quarter."

# After: chunk carries context
chunk = """[Source: ACME Corp Q2 2025 Earnings Report]
[Section: Financial Highlights]
[Date: July 2025]
Revenue grew 3% over the previous quarter."""
💡 Tip

This is the cheapest, highest-ROI improvement you can make. Zero LLM calls, zero latency impact. Just string concatenation at index time. Do this before anything else.

🔁
Iterative / Multi-Hop Retrieval

For complex questions, one retrieval pass isn't enough. Retrieve, read, formulate a follow-up query based on what you learned, retrieve again. Like "zooming in" — first pass finds the right area, second pass finds the exact answer.

# Multi-hop: "What regulations affect the drug that treats
#           the disease mentioned in Dr. Smith's latest paper?"
# Hop 1: Find Dr. Smith's paper → disease name
# Hop 2: Find drugs for that disease → drug name
# Hop 3: Find regulations for that drug → answer
📝
Query Routing

Not all queries need the same retrieval strategy. A lightweight classifier routes each query to the appropriate pipeline: simple queries → fast retrieval, complex queries → multi-hop, factual queries → keyword-heavy, exploratory queries → semantic-heavy.

💡 Tip

You can implement routing with a simple LLM call: "Classify this query as SIMPLE, COMPLEX, or ANALYTICAL." Then branch to different retrieval configurations. This saves cost on simple queries and improves quality on complex ones.

💾
Retrieval-Based Memory

Store conversation history in the same vector store as documents. When a user asks a follow-up, the system retrieves relevant prior messages alongside document chunks, enabling contextual multi-turn conversations without stuffing the entire history into the prompt.

📄
Summary-Level Indexing

Pre-compute summaries of each document (or section) and index those summaries alongside the detailed chunks. Broad queries hit summaries first, then drill into detail chunks for specific answers.

💡 Tip

This technique pairs perfectly with parent-document retrieval. Summary → full section → specific chunk forms a three-level hierarchy that handles everything from "give me an overview" to "what's the exact number."

🔗
Citation & Grounding

Force the LLM to output inline citations pointing to specific chunks. Then verify those citations by checking if the cited chunk actually supports the claim. This is the foundation of trustworthy RAG.

# System prompt for citation
"""When answering, cite your sources using [1], [2], etc.
Each citation must reference a specific chunk from the context.
If a claim cannot be supported by the context, do not make it.
At the end, list all cited sources with their chunk IDs."""

07 — EVALUATIONMeasuring What Matters

LLMs are non-deterministic. The same prompt can produce different outputs across runs. Without proper evaluation, you're flying blind.

The Two Halves of RAG Evaluation

Retrieval Quality

Precision@k: Of the k retrieved documents, how many are relevant? Measures noise — are you fetching junk?

Recall@k: Of all relevant documents in the corpus, how many did you retrieve? Measures coverage — are you missing anything?

MRR (Mean Reciprocal Rank): How high does the first relevant result appear? Position matters — rank 1 is worth 10x rank 5.

Generation Quality

Faithfulness: Does the answer actually follow from the retrieved context? Catches hallucination where the LLM invents unsupported claims.

Answer Relevance: Does the answer address the question? A factually correct but off-topic answer still fails the user.

Context Relevance: Are the retrieved chunks relevant to the question? Isolates whether poor answers stem from bad retrieval or bad generation.

The RAGAS Framework

RAGAS (Retrieval Augmented Generation Assessment) is the de facto standard for end-to-end RAG evaluation. It measures four dimensions without requiring ground-truth labels:

MetricWhat It MeasuresHow
FaithfulnessIs the answer grounded in the context?LLM checks if each claim has supporting evidence
Answer RelevanceDoes the answer address the question?Generates questions from the answer, checks similarity to original
Context PrecisionAre relevant chunks ranked higher?Checks ordering of useful vs. useless context
Context RecallIs all needed info retrieved?Checks if ground truth answers are covered by context
💡 Tip

Run RAGAS as a CI/CD gate. Before deploying any RAG pipeline change, automatically run your eval suite against a held-out set of question-answer pairs. If faithfulness drops below threshold, block the deploy.

Building Your Eval Set

Start with 50-100 real user queries from logs. Synthetic queries miss the patterns real users create.
Include adversarial cases: queries that should return "I don't know," queries with negation, multi-part questions, queries using domain jargon vs. plain language.
Tag queries by difficulty: simple factoid, multi-hop reasoning, comparison, summarization. Track metrics per category to find weaknesses.
Version your eval set. As your corpus changes, update your test questions to match. Stale evals give false confidence.
Blend automated + human evaluation. RAGAS catches systematic issues; human review catches nuance, tone, and "feels wrong" answers that metrics miss.

08 — PRODUCTION PLAYBOOK30 Battle-Tested Tips

Hard-won lessons from teams running RAG in production at scale. Organized by pipeline stage.

Tip #1

Clean your data before anything else. Garbage in, garbage out applies tenfold to RAG. Remove duplicate documents, fix encoding issues, strip boilerplate headers/footers. This alone can improve retrieval quality by 20%+.

Tip #2

Extract and index metadata aggressively. Date, author, department, document type, version number, language, section headings. Rich metadata enables pre-filtering that massively reduces your search space.

Tip #3

Chunk configuration is per-import, not per-corpus. You can re-import the same files with different chunking to test what works. Don't lock yourself into one strategy — GCP's Vertex team recommends treating chunk size as a tunable parameter.

Tip #4

Use overlap between chunks. 10-20% overlap ensures that information at chunk boundaries isn't lost. For a 512-token chunk, use 50-100 tokens of overlap.

Tip #5

Prepend contextual headers to every chunk at index time. Document title, section heading, date. Zero cost, massive impact. Do this before any fancy techniques.

Tip #6

Build both a vector index and a BM25 index. BM25 indexing is free (local computation). The two together consistently outperform either alone.

Tip #7

Set up an incremental ingestion pipeline. Don't re-index everything when one document changes. Track document hashes, only re-embed what's new or modified.

Tip #8

Retrieve more, then filter down. Retrieve top-50 to top-150 candidates, then rerank to get top-5 to top-20. Casting a wider net first ensures you don't miss relevant results that would have been cut off at top-10.

Tip #9

Always add a reranker. Even a simple cross-encoder reranker on top of hybrid search measurably improves answer quality. It's the single best post-retrieval improvement.

Tip #10

Set a distance threshold. Don't return results below a minimum similarity score. It's better to return "I don't have information about this" than to return low-relevance noise that causes hallucination.

Tip #11

Tune your alpha (dense vs. sparse weighting). Start at 0.5, then test 0.3, 0.5, 0.7 on your eval set. The optimal balance depends on your query patterns. Code-heavy? Bias sparse. Conversational? Bias dense.

Tip #12

Use query transformation as a first resort, not a last. A rewritten query costs one cheap LLM call but can make the difference between retrieving the right documents and retrieving nothing useful.

Tip #13

Implement query routing for mixed workloads. Route simple queries to fast pipelines and complex queries to multi-hop pipelines. You save cost and latency on the easy stuff while investing in quality for the hard stuff.

Tip #14

Use embedding models matched to your domain. General-purpose embedding models are fine for general content. For legal, medical, or code domains, fine-tuned embeddings consistently outperform.

Tip #15

Put the most relevant chunk first in context. LLMs attend most to the beginning. After reranking, your best result should lead the context window.

Tip #16

Use structured context formatting. Wrap each chunk in XML-like tags with metadata: <source doc="policy.pdf" page="3" section="Refunds">...</source>. This helps the LLM attribute information correctly.

Tip #17

Always provide an "I don't know" escape. Explicitly instruct the model: "If the provided context does not contain enough information to answer, say so rather than guessing."

Tip #18

Force citation in the generation prompt. "Cite your sources using [1], [2]" enables verification and builds trust. Users can click to verify.

Tip #19

Compress context before generation. For large chunks, extract only the relevant sentences before passing to the LLM. Less noise = sharper answers + lower token costs.

Tip #20

Match temperature to the task. Factual Q&A → 0.0. Summarization → 0.3. Creative synthesis → 0.5. Never use high temperature for grounded RAG responses.

Tip #21

Add a self-check pass for critical applications. After generation, ask the LLM: "Does this answer contain claims not supported by the context?" It catches many hallucinations.

Tip #22

Log everything. For every query: the raw query, transformed query, retrieved chunks (with scores), reranking results, final context sent to LLM, and the generated answer. You need this for debugging and evaluation.

Tip #23

Monitor retrieval quality in production. Track the average similarity score of top-k results over time. A declining average signals corpus drift or emerging query patterns you haven't optimized for.

Tip #24

Set up automated eval as a CI/CD gate. Run your eval suite before every pipeline change. If faithfulness or relevance drops below threshold, block the deploy.

Tip #25

Version your embedding model. When you switch embedding models, you must re-embed your entire corpus. Plan for this — it's expensive but sometimes necessary.

Tip #26

Set corpus refresh schedules. Stale documents produce stale answers. For fast-moving domains, re-ingest daily. For stable corpora, weekly or on-change is fine.

Tip #27

Implement user feedback loops. Thumbs up/down on answers, with the query and retrieved context attached. This becomes your gold evaluation dataset over time.

Tip #28

Budget for latency by pipeline stage. Know how much each step costs: embedding (5ms), vector search (20ms), reranking (200ms), LLM generation (1-5s). This tells you where to optimize.

Tip #29

Use prompt caching for repeated context. If many queries hit the same documents, prompt caching (available from Anthropic, OpenAI, etc.) can cut generation costs significantly.

Tip #30

Start simple, measure, improve iteratively. The progression: Naive → Hybrid search → Reranking → Contextual headers → Query transformation → CRAG/Self-RAG. Each step should be validated by your eval suite before moving to the next.

The "BM25 First" Philosophy

🎯
Build Search Foundations Before Adding AI

The most common mistake in RAG engineering: jumping straight to vector search and skipping keyword search entirely. Production teams at successful companies do the opposite — they master BM25 keyword search first, prove it works, then layer semantic vectors on top.

Why this matters: If your BM25 search can't find the right document, your vector search probably can't either — and now you've added complexity, cost, and latency without solving the core problem. BM25 catches exact terms, identifiers, and structured queries that dense vectors routinely miss.

Agentic RAG: The 6-Node State Machine

The arXiv Paper Curator's Week 7 implements agentic RAG not as a vague "agent that retrieves," but as a concrete LangGraph state machine with 6 specialized nodes and conditional edges that enable self-correction loops.

# LangGraph Agentic RAG — the actual architecture
# Key insight: LangGraph supports CYCLES (loops back),
# which is impossible in plain LangChain chains.

# Node 1: GUARDRAIL
# → Is this query in-domain? Score relevance to your corpus.
# → Out-of-scope queries exit immediately (saves $$)

# Node 2: RETRIEVE
# → Hybrid search (BM25 + vectors via OpenSearch)

# Node 3: GRADE DOCUMENTS
# → LLM evaluates: "Do these docs actually answer the question?"
# → Scores each chunk as relevant / not relevant

# Node 4: REWRITE QUERY (conditional)
# → If grading fails, rewrite query and LOOP BACK to retrieve
# → This is the self-correction cycle

# Node 5: GENERATE
# → Synthesize answer from graded, relevant documents

# Node 6: ERROR HANDLING
# → Graceful degradation when all retrieval attempts fail

# The flow:
Query → Guardrail ──┬─ in-domain ──→ Retrieve → Grade
                   │                              │
                   └─ off-topic → Exit         relevant? 
                                                 │    │
                                                Yes    No
                                                 │    │
                                            Generate  Rewrite → Retrieve (loop)
                                                       ↑_______________|
                                                    (max 2 attempts)
💡 Why LangGraph, Not LangChain

LangChain chains are DAGs — data flows forward only. LangGraph supports cycles: a node can route execution back to an earlier node. That loop from "Grade → Rewrite → Retrieve" is impossible in a chain. It's a single conditional edge in a graph. This is what makes the system genuinely self-correcting.

⚠ Production Guard Rails

Always cap your retry loops. The arXiv curator caps at 2 retrieval attempts. Without this cap, an unanswerable query creates an infinite loop that burns tokens and time. When retries exhaust, the error handling node returns "insufficient information" rather than hallucinating.

# Cache key strategy: hash of (query + search_mode + top_k)
# NOT the embedding — the raw query string is the key
# TTL: 1 hour for fast-changing corpora, 24h for stable ones

import hashlib, json, redis

def cache_key(query, search_mode, top_k):
    raw = json.dumps({"q": query, "m": search_mode, "k": top_k})
    return f"rag:{hashlib.sha256(raw.encode()).hexdigest()}"

# On query: check cache first, skip entire pipeline if hit
# On miss: run pipeline, store result with TTL
# On corpus update: invalidate relevant cache keys
💡 When to Cache

Exact-match caching works when your users ask similar questions repeatedly — which is true for most customer support, FAQ, and internal knowledge base use cases. It's less useful for exploratory research queries where every question is unique. Always implement cache invalidation tied to corpus updates.

Knowledge Check

1. Your RAG system returns the right document but the LLM's answer contradicts it. What's the most likely root cause?
Correct! The "lost in the middle" problem causes LLMs to pay less attention to content in the middle of the context window. The fix: use a reranker to put the most relevant chunks first, and consider context compression to remove noise.
2. Your users query with exact product IDs but vector search returns irrelevant results. What should you add first?
Correct! Dense vector search weakens when queries depend on exact terms like product IDs, ticket numbers, or error codes. These terms lose priority in embedding space. BM25 excels at exact-match retrieval, and hybrid search gives you the best of both.
3. What is Anthropic's "Contextual Retrieval" technique?
Correct! Contextual Retrieval uses an LLM to generate a short contextual explanation for each chunk that situates it within the full document. Combined with contextual BM25 and reranking, it reduces retrieval failures by 67%.
4. When does HyDE (Hypothetical Document Embeddings) fail?
Correct! HyDE relies on the LLM generating a plausible hypothetical answer. If the LLM has no knowledge about the topic, it generates a wrong or irrelevant hypothetical document, which then retrieves wrong results. Always test with out-of-domain queries and add a fallback to standard embedding.
5. What's the recommended production progression for improving a RAG system?
Correct! Start simple, measure, improve iteratively. Each step should be validated by your eval suite before adding the next layer of complexity. This way you know exactly which improvement moved the needle.
6. In an agentic RAG state machine, what happens when the Document Grader scores retrieved chunks as "not relevant"?
Correct! The self-correction cycle is the core of agentic RAG: Grade → Rewrite → Retrieve again. This loop is what makes LangGraph essential — LangChain chains can't loop back. Always cap retries (typically 2 attempts) to prevent infinite loops on unanswerable queries.
